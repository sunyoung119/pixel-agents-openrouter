0. 이 노트가 답하는 것

한 문장. "AI가 일하는 걸 픽셀 캐릭터로 보여주는 프로그램(Pixel Agents)"이 원래는 Claude 하나만 보여줬는데, 내가 서버를 한 줄도 안 고치고 GPT·Gemini 같은 다른 모델도 캐릭터로 등장시키고, 글자를 한국어로 바꾸고, 캐릭터 그림까지 갈아끼운 과정입니다.

가장 신기하고 중요한 부분은 "서버를 안 고쳤다"는 점이에요. 남이 만든 프로그램을 뜯어고치지 않고도 기능을 붙일 수 있었던 이유 — 그게 이 노트의 진짜 주제입니다.

1. 준비운동 — 원래 프로그램이 뭐하는 거였나

Pixel Agents는 AI 코딩 도구(Claude Code)가 일하는 모습을, 도트 그래픽 사무실에서 캐릭터가 타이핑하고 책을 읽는 애니메이션으로 보여주는 프로그램이에요. 여기서 딱 하나만 기억하면 됩니다.

Pixel Agents는 "관찰형"이다. 스스로 AI를 실행하지 않아요. AI가 "나 지금 이 파일 읽어요"라고 보내주는 소식(이벤트)만 받아서 그림을 그립니다.

비유하면 관제실 화면이에요. 현장 작업자가 무전으로 "나 지금 3층 점검 중"이라고 하면, 관제 화면의 내 아이콘이 3층으로 움직이는 거죠. 관제 화면은 직접 현장에 가지 않아요. 무전만 듣습니다.

핵심 포인트. 관제 화면은 무전 내용의 형식(규격)만 맞으면 그게 누구 무전인지 안 따집니다. 이 성질이 나중에 "다른 모델도 띄우기"를 가능하게 하는 열쇠예요. 지금은 이 한 줄만 기억하세요.

그런데 원래 Pixel Agents에서 이 무전을 칠 줄 아는 건 Claude Code 하나뿐이었어요. Claude Code는 실행되면 자동으로 "나 파일 읽기 시작", "다 읽음" 같은 무전을 서버로 보내도록 만들어져 있거든요.

2. 목표 — 뭘 하고 싶었나

나는 GPT, Gemini, DeepSeek 같은 다른 모델도 이 사무실에 캐릭터로 띄우고 싶었어요. 그런데 바로 문제에 부딪힙니다.

문제. GPT·Gemini에는 Claude Code 같은 "자동으로 무전 치는 기능"이 없습니다. 게다가 Pixel Agents는 관찰형이라 그 모델들을 스스로 들여다볼 수도 없어요. 무전이 안 오면 화면엔 아무 캐릭터도 안 뜹니다.

여기서 1장의 열쇠가 등장합니다. 서버는 무전 규격만 맞으면 누가 보냈는지 안 따진다. 그러면 답은 간단해요.

큰 그림 해법. "GPT를 실행하면서, 그 활동을 무전 규격에 맞춰 대신 보내주는 프로그램"을 하나 만들면 된다. 이 대신 보내주는 프로그램을 브리지(bridge)라고 부를게요. 파일 이름은 openrouter-agents/run.mjs입니다.

이제부터 이걸 다섯 단계로 나눠서, 각 단계마다 원래 → 바꾼 것 → 결과를 봅니다. 쉬운 것부터 시작해요.

바꾼 파일 지도 (미리 한눈에)

단계바꾼/만든 파일한 일
1server/.../claude.ts캐릭터 위 글자를 한국어로 (문구만)
2·3·4openrouter-agents/run.mjs 신규다른 모델을 실행하며 무전을 대신 보내는 브리지
5webview-ui/.../characters/캐릭터 그림(스프라이트) 교체

보다시피 pixel-agents의 서버·핵심 로직 파일은 안 건드립니다. 새로 만든 건 브리지 하나뿐이고, 나머지는 글자와 그림 교체예요.

1

말부터 한국어로

난이도 ★☆☆

가장 쉬운 워밍업. 캐릭터 머리 위에 뜨는 글자만 한국어로 바꿉니다.

원래는캐릭터 위 글자가 영어였어요. Reading foo.ts 처럼.
이렇게 바꿈claude.ts의 formatToolStatus() 함수에서 문구만 한국어로 교체.
그래서foo.ts 읽는 중 처럼 한국어로 표시돼요.
// server/src/providers/hook/claude/claude.ts — formatToolStatus()
// [원래]
case 'Read':  return `Reading ${base(inp.file_path)}`;
case 'Grep':  return 'Searching code';

// [바꿈]
case 'Read':  return `${base(inp.file_path)} 읽는 중`;
case 'Grep':  return '코드 검색 중';
여기서 배우는 것 — 표현과 로직의 분리. "보여주는 글자"를 만드는 부분이 이 함수 한 곳에 모여 있어요. 그래서 기능(로직)은 안 건드리고 글자만 안전하게 바꿀 수 있었죠. 프로그램이 이렇게 나뉘어 있으면 초보자도 안심하고 고칠 수 있습니다.
2

다른 모델을 무대에 올리기 (브리지)

난이도 ★★★ 핵심

이 단계가 프로젝트의 심장이에요. GPT를 사무실에 등장시키는 부분.

원래는Claude Code만 자동으로 서버에 무전(이벤트)을 보냈어요. GPT는 무전 기능이 없어 화면에 안 떴죠.
이렇게 바꿈run.mjs라는 새 프로그램을 만들어, GPT를 돌리면서 그 활동을 무전 규격에 맞춰 서버로 대신 전송.
그래서GPT가 사무실에 캐릭터로 등장! 서버는 무전 규격이 맞으니 그냥 받아줍니다.

브리지가 보내는 무전은 Claude가 보내던 것과 똑같은 모양이에요. 그래서 서버에 이미 있던 "무전 받는 창구"를 그대로 씁니다. 무전 하나는 이렇게 생겼어요.

// 캐릭터 등장 → 도구 사용 → 완료 → 퇴장  (Claude가 보내던 것과 같은 규격)
POST /api/hooks/claude   Authorization: Bearer <token>
{ "session_id": "orl-gpt-4o-mini-ab12", "hook_event_name": "SessionStart" }
{ "session_id": "orl-gpt-4o-mini-ab12", "hook_event_name": "PreToolUse",
  "tool_name": "Read", "tool_input": { "file_path": "data.json" } }
{ "session_id": "orl-gpt-4o-mini-ab12", "hook_event_name": "Stop" }
초보자 풀이 — "무전 받는 창구"가 뭐예요? 서버가 열어둔 주소예요 (POST /api/hooks/…). 프로그램이 서로 데이터를 주고받는 우편함이라고 생각하면 됩니다. 서버는 이 우편함에 규격 맞는 편지가 오면 캐릭터를 움직여요. 편지를 누가 넣었는지는 안 물어봐요. 그래서 브리지가 대신 넣어도 통하는 거죠.
여기서 배우는 것 — 계약(contract) 기반 통합. 남의 프로그램을 뜯어고치는 대신, 그 프로그램이 공개해 둔 규칙(우편함 주소 + 편지 형식)만 지켜서 데이터를 넣었어요. 그래서 서버 코드를 한 줄도 안 고치고 새 기능이 붙었습니다. 이게 "코어 무수정"의 정체예요.
3

캐릭터가 '진짜로' 일하게

난이도 ★★☆

단계 2로 캐릭터는 떴어요. 그런데 그 캐릭터가 진짜 일을 하는지, 연기를 하는지가 갈리는 지점입니다.

순진한 방법"읽기 → 쓰기 → 끝" 순서를 미리 짜서 재생. 그럴듯하지만 모델이 실제로 뭘 하는지와 무관한 가짜 연기.
이렇게 바꿈모델에게 진짜 도구(파일 읽기·쓰기)를 쥐여주고(function calling), 모델이 스스로 고른 도구를 그대로 무전으로 보냄.
그래서모델마다 일하는 순서가 다릅니다. 그리고 실제로 summary.md 파일을 만들어요. 진짜니까요.
// openrouter-agents/run.mjs — 에이전트 루프 (단순화)
while (step++ < MAX_STEPS) {
  const msg = await chat(model, messages, TOOLS);   // 모델에게 물어봄
  if (!msg.tool_calls) { hook(Stop); break; }    // 도구 안 부르면 = 일 끝
  for (const call of msg.tool_calls) {          // 모델이 '실제로' 부른 도구
    hook(PreToolUse, TOOL_MAP[call.name]);       // → 캐릭터가 그 활동을 함 (무전)
    const result = executeTool(call.name, call.args); // 진짜로 실행
    hook(PostToolUse);
    messages.push({ role: 'tool', content: result }); // 결과를 모델에게 돌려줌
  }
}
초보자 풀이 — function calling이 뭐예요? 모델은 스스로 파일을 열 손이 없어요. 대신 "이 도구 좀 써주세요"라고 요청할 수 있게 도구 목록을 줍니다. 모델이 "read_file(data.json) 불러줘"라고 하면, 내 코드가 실제로 파일을 읽어서 결과를 모델에게 돌려줘요. 이 물어보기 → 도구 실행 → 결과 돌려주기의 반복이 바로 "에이전트"입니다.
여기서 배우는 것 — 연출과 실제의 차이. 미리 짠 순서를 재생하면 데모는 되지만 거짓이에요. 실제 도구 호출을 그대로 보여주면, 모델마다 다른 진짜 행동이 화면에 나타납니다. "보여주기"가 아니라 "실제로 시키기"가 이 프로젝트가 진짜인 이유예요.
4

캐릭터에 모델 이름표 붙이기

난이도 ★★☆

모델들이 떴는데, 처음엔 이름이 다 똑같았어요. 이걸 각자 모델명으로 바꾼 이야기.

문제캐릭터 이름이 전부 pixel-agents-main으로 똑같이 나왔어요. 누가 GPT고 누가 Gemini인지 구분 불가.
원인 추적 → 바꿈서버는 캐릭터 이름을 '작업 폴더의 마지막 이름'으로 정하더라고요. 그래서 각 모델이 자기 모델명 폴더에서 일하게 함.
그래서폴더 이름 = 모델 이름 = 캐릭터 이름. gpt-4o-mini, gemini-2.5-flash로 딱 구분돼요.
// 서버가 이름을 정하는 방식 (원본 코드, 안 고침)
const folderName = cwd ? path.basename(cwd) : undefined;   // ← 캐릭터 이름

// 그래서 브리지에서 각 모델의 작업 폴더를 '모델명'으로 지정
cwd: `${WORKSPACE_ROOT}/gpt-4o-mini`   // → 캐릭터 이름 = "gpt-4o-mini"
여기서 배우는 것 — 시스템의 규칙을 이해하고 맞추기. 이름이 이상하게 나올 때 서버를 고치려 들지 않았어요. 대신 "서버가 어떤 규칙으로 이름을 정하나?"를 알아냈고 (폴더명이었죠), 그 규칙에 맞게 입력 데이터를 조정했습니다. 남의 프로그램과 협력하는 정석이에요.
5

캐릭터 그림 바꾸기

난이도 ★★☆

마지막으로 캐릭터 그림을 갈아끼웁니다. (네가 직접 소방관으로 바꿔본 그 작업이에요.)

원래는기본 제공 캐릭터 6종.
이렇게 바꿈char_0.png~char_5.png를 규격에 맞는 커스텀 그림으로 덮어쓰기.
그래서내가 원하는 캐릭터가 사무실에 등장. (코드는 0줄 수정.)

여기서 중요한 건 아무 그림이나 넣으면 안 된다는 점이에요. 렌더 엔진이 기대하는 정확한 규격이 있어요.

항목규격
파일char_0.png ~ char_5.png (6개)
크기112 × 96 픽셀
배치가로 7프레임(16px) × 세로 3방향(32px) = 21칸. 행0=정면, 행1=뒤, 행2=오른쪽 (왼쪽은 자동 반전)
프레임 순서걷기1·걷기2·걷기3 · 타이핑1·타이핑2 · 읽기1·읽기2
필수배경 투명 · 21칸 모두 발 높이(바닥선) 동일
초보자 풀이 — 스프라이트 시트가 뭐예요? 한 장의 그림을 격자로 잘라 여러 장면을 뽑고, 그걸 빠르게 순서대로 보여주면 움직이는 것처럼 보여요(만화 플립북이랑 같아요). 그래서 아무렇게나 그리면 안 되고, 격자 칸과 발 높이를 정확히 맞춰야 걷기·앉기가 자연스럽습니다. 발 높이가 칸마다 다르면 걸을 때 캐릭터가 위아래로 통통 튀어요.
여기서 배우는 것 — 규격이 곧 계약. 단계 2의 "무전 규격"과 똑같은 원리예요. 그림도 정해진 규격(격자·크기·발높이)만 지키면 코드를 안 고치고 교체됩니다. 프로그램의 여러 부분이 "정해진 약속"으로 연결돼 있다는 걸 그림에서도 다시 확인하는 거죠.

전체를 한 그림으로

다섯 단계를 하나로 이으면 이렇게 흘러요.

챙겨갈 개념 5가지 (아주 쉽게)

1. 관찰형. "보여주는 프로그램"과 "실제로 실행하는 프로그램"은 다를 수 있다. Pixel Agents는 보여주기만 하고, 실행은 브리지가 했다.
2. 계약(약속) 기반 통합. 남의 프로그램을 안 고쳐도, 그 프로그램이 정해둔 약속(우편함 주소 + 편지 형식 + 이름 규칙)만 지키면 새 기능을 붙일 수 있다.
3. 정규화(번역 문지기). 제각각으로 들어온 무전을 서버가 하나의 표준 형태로 바꾼다. 그래서 뒷단은 무전을 누가 보냈든 신경 안 써도 된다.
4. 에이전트 루프. "모델에게 물어보기 → 도구 실제 실행 → 결과 돌려주기"의 반복. 이 반복이 AI 에이전트의 본질이다.
5. 표현/로직 분리. 글자·그림 같은 "겉모습"이 기능과 나뉘어 있으면, 기능을 안 건드리고도 안전하게 바꿀 수 있다. (한국어화·캐릭터 교체가 쉬웠던 이유.)
